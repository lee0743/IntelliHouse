### Q1. 투자성향 부적합 — 강제 차단이 아닌 이유

가입자의 투자성향과 선택 상품의 위험등급이 불일치할 때, 시스템이 강제로 차단하지 않고 확인 절차만 거치는 이유를 설명하시오.

<details>
<summary>정답 보기</summary>

퇴직연금은 **자기결정권과 책임 부여** 원칙을 따른다. 투자성향 검사는 가입자가 위험을 인지했음을 확인하는 절차이지, 선택을 금지하는 장치가 아니다.

- 강제 차단 시: 가입자의 투자 자유 침해, 고위험 상품을 원하는 가입자도 불가
- 확인 절차 방식: 위험 인지 후 가입자 본인이 선택 → **운용 결과는 가입자 책임**

시스템 설계 관점에서는 경고 팝업 → 가입자 확인(동의) → 지시 수용의 흐름으로 구현된다.

</details>

### Q2. 운용관리기관·자산관리기관 역할 분리의 목적과 겸영 허용의 모순 해소

두 기관의 역할 분리 목적을 설명하고, 동일 금융기관이 두 역할을 겸영할 수 있음에도 분리 구조가 유지되는 이유를 서술하시오.

<details>
<summary>정답 보기</summary>

**분리 목적**: 적립금의 **횡령·유용 방지**
- 운용관리기관이 운용지시를 내리되 실제 자산을 보관하지 않음
- 자산관리기관이 자산을 보관하되 운용 결정권이 없음
→ 한 기관이 운용 + 보관을 동시에 하면 가입자 모르게 자금 유용 가능

**겸영 허용과의 관계**: 겸영이 허용되더라도 **내부에서 역할은 반드시 분리**되어야 한다(업무 단위 분리). 법령은 역할의 혼용을 금지할 뿐 동일 법인의 겸영을 금지하지는 않는다. 다만 이해충돌 리스크가 높아져 감독 강화가 필요하다.

</details>

### Q3. 컷오프 타임과 기준가 리스크

가입자가 오후 4시에 특정 펀드 매수 운용지시를 넣었다. 어느 날의 기준가가 적용되는가? 이것이 가입자에게 어떤 리스크를 만드는지 설명하시오.

<details>
<summary>정답 보기</summary>

**적용 기준가**: 익영업일 기준가
- 오후 3시 이후 접수분은 다음 영업일 기준가로 처리

**리스크**:
- 지시 시점과 기준가 적용 시점 사이에 시장이 변동할 수 있음
- 오후 4시에 지시 → 당일 종가 기준이 아닌 **다음날 종가** 기준 적용
- 급등장에서는 더 비싸게 매수, 급락장에서는 더 싸게 매수될 수 있음
- 가입자가 컷오프 타임을 모르면 예상치 못한 체결가로 손해 볼 수 있음

</details>

### Q4. 운용오케스트라 처리 실패(5-2-1) 시 흐름과 가입자 영향

운용오케스트라 처리가 실패했을 때 시스템이 어떻게 대응하는지 서술하고, 가입자 입장에서의 영향을 설명하시오.

<details>
<summary>정답 보기</summary>

**시스템 대응**:
- 오류 처리 루틴 실행 → 재시도 또는 담당자 알림
- 백오피스에도 실패 결과가 전달되어 독립적으로 오류 기록
- 주문이 실행되지 않으므로 결제·정산 단계로 진행 불가

**가입자 영향**:
- 운용지시가 접수됐더라도 실제 매수/매도가 이루어지지 않음
- 컷오프 타임 이전에 재지시하지 않으면 당일 체결 불가 → 익영업일 기준가 적용 위험
- SMS 통지 흐름에서 처리 실패 알림이 발송되어야 가입자가 인지 가능

</details>

### Q5. DB형에서 ALM 관점 운용전략이 필요한 이유

DB형 퇴직연금에서 ALM(자산부채관리) 관점의 운용전략이 요구되는 이유를 DC형과 비교하여 설명하시오.

<details>
<summary>정답 보기</summary>

**DB형 구조**:
- 회사(사용자)가 미래 지급해야 할 퇴직급여(부채)를 적립금(자산)으로 충당
- 부채는 근로자 임금 수준·근속연수에 연동 → 부채 규모가 변동
- 적립금 운용 수익이 부채 증가를 따라가지 못하면 **적립 부족** 발생 → 회사 추가 부담

**ALM 필요 이유**:
- 자산(적립금)의 듀레이션과 부채(퇴직급여)의 듀레이션을 맞추는 전략 필요
- 단순 수익 극대화가 아니라 **부채 대비 자산 비율 유지**가 핵심 목표

**DC형과의 차이**:
- DC형은 회사가 정해진 금액만 납부하면 책임 종료 → 부채 리스크 없음
- DC형 가입자는 수익 극대화가 목표, ALM 불필요

</details>

### Q6. 집중투자 한도: 동일 발행자 10% vs 동일 운용사 30% 기준이 다른 이유

동일 발행자는 10%, 동일 집합투자업자(운용사)는 30%로 한도가 다른 이유를 설명하시오.

<details>
<summary>정답 보기</summary>

**동일 발행자 10%**:
- 특정 기업이나 기관이 발행한 증권에 집중되면 해당 발행자의 도산·부도 시 **원금 손실 리스크가 집중**됨
- 개별 종목 리스크를 강하게 분산시켜야 하므로 엄격한 10% 한도

**동일 운용사 30%**:
- 펀드는 이미 내부적으로 여러 종목에 분산투자되어 있음 → 발행자 리스크보다 낮음
- 동일 운용사의 여러 펀드에 분산 투자는 가능하지만, 운용사 자체의 **운용 역량 리스크·운용사 파산 리스크**를 고려해 30%로 제한
- 단일 발행자보다 완화된 기준 적용 가능

</details>

### Q7. 디폴트옵션 도입이 운용지시 처리 흐름에 추가한 변화

2023년 디폴트옵션(사전지정운용방법) 도입 이후 기존 운용지시 처리 흐름에 어떤 단계가 추가되었는지 서술하시오.

<details>
<summary>정답 보기</summary>

**기존 흐름**: 가입자 운용지시 → 제약 검토 → 주문 실행

**디폴트옵션 도입 후 추가 흐름**:
1. 가입자가 운용지시를 **4주 이상 미제출** 시 1차 안내
2. **6주 경과** 후에도 지시 없으면 사전지정운용방법(디폴트옵션 상품)으로 자동 운용 실행
3. 이후 가입자가 직접 운용지시를 내리면 **가입자 지시가 디폴트옵션보다 우선** 적용

**시스템 설계 관점**:
- 운용지시 미제출 가입자를 주기적으로 조회하는 배치 프로세스 추가 필요
- 자동 운용 실행 전 안내 SMS/Email 발송 흐름 추가
- 가입자 지시 수신 즉시 디폴트옵션 해제 로직 필요

</details>

### Q8. 백오피스 흐름이 프론트 흐름과 독립적으로 존재하는 이유

매수/매도 시스템에서 백오피스 처리 흐름이 프론트 흐름과 별도로 존재하는 설계 이유를 서술하시오.

<details>
<summary>정답 보기</summary>

**이중화 및 검증 목적**:
- 프론트는 가입자 인터페이스 처리에 집중, 백오피스는 **내부 정산·회계 처리**에 집중
- 동일 거래를 두 시스템이 독립적으로 기록 → **처리 오류 발생 시 상호 대조**로 오류 탐지 가능

**책임 분리**:
- 프론트에서 오류가 발생해도 백오피스 기록이 남아 **감사(audit)** 가능
- 규제 기관 보고용 데이터는 백오피스 기록 기준으로 생성

**가용성**:
- 프론트 시스템 장애 시에도 백오피스는 독립적으로 계속 작동 → 정산·결제 지연 방지

</details>
